# 关系知识字段说明

关系知识单元以“对象关系”为核心。元数据只保存需要稳定检索的关系级状态；具体判断的来源状态、边界和观测方式优先写在正文，不为 schema 完整度堆积空字段。

## 最小元数据

```yaml
---
id: relation-compute-heat-utilization
type: relation-knowledge
objects:
  - 算力设施
  - 热负荷
  - 冷却系统
  - 热力网络
knowledge_status: forming
learner_status: forming
source_refs:
  - PPT 第18—20页
  - PPT 第30页
  - PPT 第38—43页
linked_learning_notes:
  - 算电学习仓库/01-一次学习/2026-07-18-算力设施电热耦合与热负荷利用.md
---
```

字段含义：

- `id`：关系知识单元的稳定标识。
- `type`：固定为 `relation-knowledge`。
- `objects`：参与关系的核心对象，不列无实际作用的概念。
- `knowledge_status`：该关系当前是否仍在形成、已经稳定或已被后续理解替代。
- `learner_status`：用户对该关系的当前形成状态；不得由 Agent 根据讲解篇幅自行升级。
- `source_refs`：直接支撑当前关系的材料位置。
- `linked_learning_notes`：产生或修正该关系的一次学习事件。

## 判断来源状态

正文中的关键判断使用以下状态之一：

- `ppt_explicit`：PPT 或其他指定原始材料明确表达。
- `reasonable_inference`：能够从材料合理推导，但不是材料原句，也不表示用户已经形成该判断。
- `user_formed`：用户已经用自己的表达形成、修正、确认或应用。
- `agent_candidate`：Agent 提出但尚未得到用户表达确认。
- `superseded`：曾经记录、后来被新的材料或理解修正。

状态描述的是判断的来源和当前位置，不是统一的可信度评分。`ppt_explicit`、`reasonable_inference` 和 `agent_candidate` 都不能自动替代 `user_formed`。

## 判断边界

关系正文继续区分三类边界，但不要求填成大表：

- 可直接计算：边界一致且输入已知时可以按明确关系计算。
- 必须实测或建模：需要运行数据、设备特性或动态模型才能确定。
- 必须由业务、合同、制度或标准定义：物理量本身不能给出答案。

只有实际出现的字段和状态才写入关系单元；没有冲突、替代记录或外部核验需求时不设置空章节。
