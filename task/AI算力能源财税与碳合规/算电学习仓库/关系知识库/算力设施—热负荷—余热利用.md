# 算力设施—热负荷—余热利用

## 当前关系

```text
算力设施
→ 产生热量
→ 可回收热量
→ 可利用热量
```

这是一条仍在形成中的关系，不表示用户已经掌握全部环节，也不表示产生的热量必然能够回收、利用或出售。

## PPT 及材料明确支持

- `ppt_explicit`：算力运行消耗电力并形成数据中心余热，算力设施同时表现为电力负荷和热源。
- `ppt_explicit`：冷却系统把 IT 设备产生的热量从芯片、服务器或机柜侧转移出去；风冷、液冷及其供回液温度会影响余热回收条件。
- `ppt_explicit`：数据中心低品位余热可通过换热、热泵、吸收式或吸附式制冷等路径再利用，并可能连接建筑供热等热用户。
- `ppt_explicit`：液冷能够提高二次侧温度，为余热回收和后续温度提升创造条件；具体效果取决于所用系统和运行条件。

以上只记录 PPT 展示的关系方向。PPT 中的案例参数、性能比例和情景结果不自动视为通用工程结论。

## 用户已经形成的判断

- `user_formed`：用户接受以数据中心作为算力、电力和热力的耦合枢纽。
- `user_formed`：用户主动把“热负荷怎样利用、热量怎样储存”提出为需要继续理解的问题。

这两项表明用户已经进入该关系，但不足以把节点标为已掌握，因此 `learner_status` 仍为 `forming`。

## Agent 候选解释

- `agent_candidate`：产生热量、可回收热量和可利用热量不是同一个量，候选边界为“可利用热量不高于可回收热量，可回收热量不高于产生热量”。
- `agent_candidate`：物理热量不能直接推导为可售热量、经济收益或减排量；仍需热用户、合同、计量和核算边界。
- `reasonable_inference`：温度、流量、距离、时间匹配和热用户需求会共同决定产生的热量能否进入可利用状态，但各因素的权重需要具体系统数据。

这些判断尚未全部由用户用自己的表达形成、修正、确认或应用，不能升级为 `user_formed`。当前没有需要标为 `superseded` 的判断。

## 条件、边界和观测量

### 参与对象与作用方向

- 算力设施消耗电能并产生热负荷。
- 冷却系统消耗电力，将热量从 IT 设备侧转移到空气、冷却液或外部热汇。
- 换热器、热泵或热力网络继续改变热量的温度条件、空间位置和可达用户。
- 热用户需求反向限定什么温度、流量、时间和距离范围内的热量具有利用条件。

### 需要观测的量

- IT 功率与 IT 电量；
- 冷却功率与冷却电量；
- 冷却介质的供回水温度、流量和运行时段；
- 热用户需求、所需温度和需求时段；
- 算力设施与热用户的距离及输送条件；
- 热泵、换热器和管网的输入、输出与损失。

### 三类判断边界

- 可直接计算：边界一致且输入已知时，功率对时间积分得到电量；介质密度、比热、流量和供回水温差已知时，可以计算相应时段转移的热量。
- 必须实测或建模：实际 IT 功率、冷却功率、供回水温度与流量、动态热响应、可回收比例、热泵或换热损失、管网输送损失。
- 必须由业务、合同、制度或标准定义：什么温度和稳定性算“可利用”、由谁验收、怎样计量结算，以及是否形成收入、环境效益或合规结果。

## 知识冲突或张力

```text
IT 用电最终大部分转化为热
≠
这些热量都能够被回收、利用或出售
```

由此只形成候选研究问题，不直接形成结论：

- 在给定冷却方式和运行边界下，产生热量中有多少能够被稳定回收？
- 在给定温度、距离、时段和热用户需求下，可回收热量中有多少能够实际利用？
- 热泵、换热和输送所需的额外电量与损失，怎样改变经济和环境结果？
