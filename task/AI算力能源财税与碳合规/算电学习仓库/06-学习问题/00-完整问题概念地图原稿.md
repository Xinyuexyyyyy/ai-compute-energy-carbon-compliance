# 一、这套PPT的总问题

可以压缩为一句：

> **如何把异构的算力业务和AI任务，转换成可测量、可预测、可调度、可评价、可定价的电力—热力—碳排对象，并在服务质量、安全、成本和低碳目标之间实现协同？**

它下面其实包含五层问题：

```text
对象是什么
    ↓
数量怎样测
    ↓
系统怎样耦合
    ↓
运行怎样优化
    ↓
各主体为什么愿意参与
    ↓
如何形成标准、价格和市场
```

PPT的核心转换链是：

```text
业务或任务
→ token与时延
→ 计算量FLOPs
→ 服务器负载
→ IT功率
→ 发热与冷却负荷
→ PUE与数据中心总功率
→ 用电量
→ 碳排放
→ 成本与价格
→ 调度及交易
```

其中任何一环边界不清，后面的碳核算、成本计算和市场机制都会失真。

# 二、研究对象地图

## 1. 微观对象：设备与机柜

| 研究对象      | 主要观察变量          | 核心问题          |
| --------- | --------------- | ------------- |
| 芯片CPU/GPU | 频率、电压、利用率、功率、温度 | 计算负荷怎样变成功耗和发热 |
| 服务器       | 吞吐量、算力、负载率、功率   | 哪个负载率下算效最高    |
| 内存、存储、网络  | 读写率、占用率、I/O功率   | 非GPU功耗如何计入    |
| 机柜        | 总功率、功率密度、层间温差   | 高密计算怎样安全散热    |，
| 冷板及液冷回路   | 流量、入口温度、压降、换热量  | 冷却怎样跟随动态计算负荷  |
| 供配电设备     | 转换效率、损耗         | IT设备之外损失了多少电  |

微观研究的重点是**算力—功耗—生热—冷却—温度反馈**。现有研究常分别优化上层服务器和下层冷却系统，容易忽略温度柔性和热惯性带来的全局优化空间。 

## 2. 介观对象：单个数据中心或园区

| 研究对象      | 主要组成            | 核心问题         |
| --------- | --------------- | ------------ |
| 数据中心      | IT、供配电、制冷、备用电源  | 总能耗如何形成      |
| 算力综合能源系统  | 算力、电力、冷力、热力、储能  | 多种能源怎样联合优化   |
| 可再生能源供能系统 | 光伏、风电、公共电网      | 波动绿电怎样匹配稳定负荷 |
| 储能系统      | 电池、氢储能、蓄冷等      | 怎样平衡多时间尺度波动  |
| 余热利用系统    | 热泵、换热器、吸收式制冷    | 低品位余热能否形成收益  |
| 园区运营主体    | 数据中心、电网、热用户、能源商 | 成本与收益如何分配    |

## 3. 宏观对象：跨区域网络和市场

| 研究对象     | 主要组成        | 核心问题         |
| -------- | ----------- | ------------ |
| 多数据中心集群  | 不同地区的算力设施   | 任务应在哪执行      |
| 全国一体化算力网 | 算、存、运、网络    | 算力如何跨区域调度    |
| 电力系统     | 电源、电网、负荷、储能 | 算力负荷如何与电网互动  |
| 算力市场     | 供应商、平台、用户   | 算力如何描述、交易和结算 |
| 电力和碳市场   | 电价、绿电、绿证、碳价 | 外部价格如何影响任务调度 |
| 政府与标准机构  | 政策、术语、评价、监管 | 如何建立共同口径     |

PPT后部把系统进一步区分为宏观、介观、微观算力综合能源系统：宏观偏算力网与电力网，介观偏产消型数据中心，微观偏IT设备和冷却设备的全栈联动。

# 三、核心概念节点

下面不是完整教科书定义，而是它们在这张研究地图中的作用。

## A. 任务与算力层

| 概念        | 最小定义                         | 在逻辑链中的作用          |
| --------- | ---------------------------- | ----------------- |
| **业务**    | 用户真正希望完成的功能                  | 决定质量、准确率、时延和可靠性要求 |
| **任务**    | 可被计算系统执行和调度的工作单元             | 调度、计量和定价的基本候选对象   |
| **在线任务**  | 对响应时延敏感的交互任务                 | 通常难以长时间延迟         |
| **离线任务**  | 可在一定期限内完成的批处理任务              | 通常具有更强时间灵活性       |
| **token** | 模型处理文本的离散单位                  | 描述输入输出规模，但不直接等于能耗 |
| **预填充**   | 对输入上下文进行并行处理的阶段              | 计算特性与输出解码不同       |
| **解码**    | 逐步生成输出token的阶段               | 时延和功耗受输出长度影响明显    |
| **FLOPs** | 完成一次任务所需的浮点运算量               | 描述计算工作量           |
| **FLOPS** | 单位时间内的浮点运算能力                 | 描述计算性能，不等于总计算量    |
| **算力**    | 系统完成计算任务的能力                  | 需说明精度、硬件和测试口径     |
| **算效**    | 单位功率或能耗所获得的计算能力              | 连接算力与电力           |
| **任务灵活性** | 任务可延迟、中断、降级或迁移的程度            | 决定算电协同可调度空间       |
| **任务画像**  | 对任务类型、时延、规模、硬件需求和可迁移性进行结构化描述 | 是调度和交易的入口         |

token和FLOPs之间没有固定换算关系，它还受模型结构、输入长度、预填充与解码、批处理和推理框架影响。

## B. 电力与热力层

| 概念         | 最小定义                  | 在逻辑链中的作用           |
| ---------- | --------------------- | ------------------ |
| **服务器负载率** | 服务器实际使用程度             | 影响功率、吞吐量和单位算力能耗    |
| **IT功率**   | 服务器、网络、存储等IT设备的瞬时耗电速率 | 从计算进入电力系统的接口       |
| **电量**     | 功率在时间上的积分，通常用kWh      | 碳排和电费核算的基础         |
| **基线功耗**   | 服务器空闲或维持运行时的功耗        | 说明低利用率不等于低单位能耗     |
| **动态功耗**   | 随负载、频率和活动状态变化的功耗      | 用于描述任务引起的增量功耗      |
| **热负荷**    | IT设备产生、需由冷却系统移除的热量    | 连接电力与冷却            |
| **PUE**    | 数据中心总能耗除以IT设备能耗       | 将IT电量扩展为设施总电量      |
| **CUE**    | 单位IT能耗对应的碳排强度指标       | 反映能源结构与设施效率的综合影响   |
| **WUE**    | 单位IT能耗对应的用水指标         | 评价数据中心水资源效率        |
| **冷却功率**   | 移除IT设备热量所消耗的电功率       | PUE的重要组成部分         |
| **液冷**     | 以液体作为主要热量传递介质的冷却方式    | 适应高功率密度机柜          |
| **两相液冷**   | 冷却介质通过相变吸热            | 提高换热能力，但控制更复杂      |
| **温度柔性**   | 在安全允许范围内动态调整运行温度      | 提供IT与冷却联合优化空间      |
| **热惯性**    | 温度对负荷变化不是瞬时响应         | 使短时间调节成为可能，也增加控制难度 |
| **余热回收**   | 将服务器排出的热量用于供暖或制冷      | 使数据中心从单纯消费者变成产消者   |

PUE只能说明设施层效率，不能单独说明模型效率、服务器算效或碳效率。

## C. 能源、碳和经济层

| 概念          | 最小定义                          | 在逻辑链中的作用        |
| ----------- | ----------------------------- | --------------- |
| **电网排放因子**  | 单位电量对应的温室气体排放                 | 将电量转换为运行碳排      |
| **平均排放因子**  | 一段时间或地区电力结构的平均碳强度             | 适合历史核算与披露       |
| **边际排放因子**  | 新增一单位负荷引起的排放变化                | 更接近增量调度影响       |
| **绿电**      | 具有可再生能源属性的电力                  | 影响市场法碳核算和采购成本   |
| **绿证**      | 可再生能源环境属性凭证                   | 不等同于物理直连供电      |
| **绿电直连**    | 数据中心与可再生能源电源建立更直接的供能关系        | 减少属性和物理供电之间的距离  |
| **可再生能源消纳** | 将风光等新能源实际利用起来                 | 算力灵活性的潜在系统价值    |
| **碳成本**     | 碳排所对应的显性或情景价格                 | 使低碳目标进入经济目标函数   |
| **CAPEX**   | 建设和设备投资成本                     | 决定重资产项目门槛       |
| **OPEX**    | 电费、维护等运行成本                    | 算电协同直接影响较多的部分   |
| **TCO**     | 全生命周期总拥有成本                    | 防止只看节电不看建设投资    |
| **激励系数K**   | PPT中用年度电力成本与年均化建设成本的关系衡量经济驱动力 | 判断电价信号能否撬动项目    |
| **需求响应**    | 用户根据电网信号调整负荷                  | 数据中心参与电力系统互动的方式 |
| **辅助服务**    | 为电网稳定提供调频、备用等能力               | 可能形成额外收益        |
| **服务违约成本**  | 延迟或中断任务造成的损失                  | 防止为了低电价无限迁移任务   |

PPT强调，电费、绿电成本、碳成本、建设成本和服务违约成本不能混成一个不说明边界的“绿色成本”；其情景分析也表明，算电协同收益未必足以覆盖重资产投入。 

## D. 协同、优化和制度层

| 概念         | 最小定义                      | 在逻辑链中的作用       |
| ---------- | ------------------------- | -------------- |
| **算电协同**   | 计算任务与电力系统在规划或运行中相互协调      | 总体研究领域         |
| **广义算电协同** | 包括能效、供电、需求响应、调度等多种算力—电力互动 | 范围较宽           |
| **狭义算电协同** | 更强调高比例绿电条件下的任务与能源匹配       | 重点关注新能源协同      |
| **算电热协同**  | 同时考虑计算、电力消耗、发热与冷却         | 单体数据中心的重要物理链   |
| **全栈联动**   | 联合优化上层IT设备和下层制冷、供配电设备     | 避免局部最优         |
| **综合能源系统** | 统一协调电、冷、热、储能和可再生能源        | 园区层研究框架        |
| **时序迁移**   | 将任务移动到另一个时间执行             | 利用电价或碳强度时间差    |
| **空间迁移**   | 将任务移动到另一个地区或数据中心          | 利用区域资源差异       |
| **日前调度**   | 根据预测提前安排次日运行              | 处理经济性和能源配置     |
| **日内控制**   | 根据实时状态修正运行                | 处理预测误差和动态安全    |
| **多目标优化**  | 同时考虑成本、碳排、能效、时延等目标        | 描述目标冲突         |
| **不确定性**   | 任务到达、风光出力、电价等不可完全预知       | 决定模型是否可靠       |
| **分布鲁棒优化** | 在概率分布不完全确定时寻找较稳健方案        | PPT技术案例采用的方法之一 |
| **多维画像**   | 用统一字段描述任务、算力和用电特征         | 连接测量、调度与市场     |
| **计量**     | 明确对象、边界、单位和测量方法           | 后续定价与监管的前提     |
| **评价体系**   | 用指标判断能效、碳效、水效和经济性         | 形成比较依据         |
| **标准体系**   | 统一术语、方法、接口和测试规则           | 使多主体能够协作       |
| **交易结算**   | 对算力、电力和环境属性进行定价、交易和收益分配   | 规模化落地的制度接口     |

PPT最后的建议正是从试点、token计费、任务画像和用电监测，逐渐推进到术语、标准、评价和多链条服务。

# 四、概念与问题的主连线

下面这张图最重要。

```text
[AI业务需求]
  │
  ├─问题：究竟要完成什么功能？
  ├─问题：质量、准确率、时延、可靠性要求是什么？
  ↓
[计算任务]
  │
  ├─问题：怎样描述任务规模？────────→ [输入/输出token]
  ├─问题：怎样描述真实工作量？──────→ [FLOPs]
  ├─问题：能否延迟、中断、迁移？────→ [任务灵活性]
  └─问题：需要何种硬件与软件？──────→ [GPU/服务器/推理框架]
  ↓
[服务器执行]
  │
  ├─问题：负载率怎样影响吞吐量？
  ├─问题：空闲功耗如何分摊？
  ├─问题：批处理怎样改变单位任务能耗？
  ↓
[IT功率]
  │
  ├───────────────────────┐
  │                       │
  │ 功率×时间             │ 大部分电功最终变成热
  ↓                       ↓
[IT电量]                [热负荷]
  │                       │
  │ ×PUE                  ├─问题：需要多少冷却功率？
  │                       ├─问题：温度可以放宽多少？
  │                       └─问题：热惯性如何利用？
  ↓                       ↓
[设施总电量] ←──────── [冷却与供配电能耗]
  │
  ├─×电价──────────────→ [电费]
  ├─×排放因子──────────→ [运行碳排]
  ├─结合绿电/绿证──────→ [绿色属性]
  └─结合CAPEX、运维────→ [TCO]
  ↓
[运行决策]
  │
  ├─问题：什么时候运行？────────→ [时序调度]
  ├─问题：在哪里运行？──────────→ [空间迁移]
  ├─问题：运行多少？────────────→ [负荷控制]
  └─问题：温度和冷量如何配合？──→ [算冷全栈控制]
  ↓
[技术效果]
  │
  ├─节能多少？
  ├─减碳多少？
  ├─降低成本多少？
  └─是否影响SLA、可靠性和寿命？
  ↓
[经济激励]
  │
  ├─谁投资？
  ├─谁节省成本？
  ├─谁获得需求响应收益？
  └─收益能否覆盖建设和迁移成本？
  ↓
[市场和制度]
  │
  ├─任务怎样画像？
  ├─用电怎样监测？
  ├─算力怎样定价？
  ├─碳成本怎样计入？
  ├─收益怎样分配？
  └─术语、测试与评价如何统一？
```

这张图中还有三个重要的**反馈回路**：

```text
温度升高
→ 芯片频率、功耗和可靠性变化
→ 计算性能变化
→ 任务运行时间变化
→ 能耗和发热再次变化
```

```text
电价、碳强度和绿电可用性变化
→ 调度策略变化
→ 任务执行时间与地点变化
→ 数据中心负荷变化
→ 电网和能源系统状态变化
```

```text
市场价格和标准
→ 影响企业决策与投资
→ 改变设备、能源和调度方案
→ 形成新的成本与性能数据
→ 反过来修正标准和价格
```

# 五、PPT能够拎出的完整问题树

## 模块一：问题是否真实存在

1. AI任务增长是否必然导致数据中心用电同比例增长？
2. 训练、推理、存储和网络分别贡献多少能耗？
3. 数据中心能耗统计为什么差异巨大？
4. 自上而下统计和自下而上估算如何校准？
5. 上架率、利用率、设备型号和PUE如何进入预测？
6. 中国面临的是全国性缺电，还是局部区域容量和供电问题？
7. 数据中心在不同城市和枢纽节点的用电压力是否不同？
8. 算力基础设施未来是否具有足够大的可调负荷规模？
9. “算力的尽头是电力”在哪些条件下成立？
10. 中美单位算力碳排差异由芯片、电网还是PUE主导？

PPT开头没有直接把算力增长等同于缺电，而是先讨论能耗统计口径、区域差异、芯片效率、PUE和能源结构。

## 模块二：任务怎样被认识和计量

11. 什么才是最合适的计量对象：请求、token、任务还是FLOPs？
12. 相同token数量是否代表相同计算量？
13. 输入token和输出token的能耗特征是否相同？
14. 预填充和解码阶段如何分别计量？
15. 如何处理不同模型的输出质量和长度差异？
16. 如何把任务类型、时延、精度和可靠性纳入画像？
17. 在线与离线任务应如何分类？
18. 如何衡量任务可延迟、可中断和可迁移程度？
19. 任务画像需要哪些最少字段？

# 六、这张地图的逻辑骨架

这120多个问题看起来很多，但本质只有六类。

```text
识别问题：
为什么需要协同？

计量问题：
对象是什么，怎样量化？

机理问题：
算力、电力和热力怎样互相影响？

优化问题：
在约束下怎样做决策？

经济问题：
技术效果能否变成主体动力？

制度问题：
如何形成统一口径、价格和交易？
```

也可以表达为一条研究推进路径：

```text
统计与监测
→ 任务和设备建模
→ 算电热耦合机理
→ 单体和跨区域优化
→ 技术效果验证
→ TCO与激励分析
→ 画像、评价和标准
→ 价格、市场与治理
```

# 七、你第一轮真正需要掌握的12个节点

整张图不能一天学完。第一轮先把下面12个节点标成主节点：

```text
1. 业务/任务
2. token
3. FLOPs
4. 任务灵活性
5. 服务器负载率
6. IT功率
7. 热负荷
8. PUE
9. 用电量
10. 排放因子
11. TCO
12. 多维任务画像
```

第一轮学习只沿这一条走：

```text
任务
→ token与时延
→ FLOPs和硬件需求
→ 服务器负载与功率
→ 发热与冷却
→ PUE和总电量
→ 碳排和成本
→ 任务画像
→ 是否具备调度条件
```

这也是目前最适合作为你学习笔记主体的结构。PPT虽然覆盖设备、园区、跨区域和市场四个层次，但对个人初步研究而言，计量应放在第一层，经济性验证放在第二层，低碳调度放在第三层。
